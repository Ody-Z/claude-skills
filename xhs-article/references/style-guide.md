# Ody 的小红书写作风格指南

Phase 3 撰写初稿时的写作规范。

三个核心：**活人感**、**读起来流畅**、**信息密度高**。三个同时成立，少一个就不是 Ody 的文。

---

## 一、活人感（像 Ody 本人在讲，不是 AI 在输出）

活人感的反面是 AI 味——信息在，但没人。读者一看就知道背后有没有一个真的人。

### 1.1 第一人称必须真实

- "我发现"、"我经历"、"我觉得"、"我当时"、"我试过"——这些话背后必须有 Ody 亲口提供或授权的素材。AI 不得自己编造
- 没素材宁可标 `[待 Ody 补充一手细节]`，不要用"比如有一次..."填空
- 情绪词（震撼/焦虑/兴奋/愣住/鼻子一酸）也要核对。体感记忆 > 知识性情绪描述
- 正面示例："我愣了一下"、"那两天我睡不着"
- 反面示例："我感到非常震撼"、"这让我陷入深思"

### 1.2 立场通过行动带出，不贴标签

个人观点要自然融入，像朋友聊天顺口说出。不能是标签式声明。

- 好："我宁愿承担亏掉 30% 资产的风险，也不愿错过这波浪潮"
- 好："每隔几个月回头看之前的做法都觉得原始"
- 坏："我的框架是：X"
- 坏："我的观察是：Y"
- 坏："我的判断是：Z"

### 1.3 姿态：谦逊观察者，不是导师/博主/营销号

- 敢下判断："我觉得 X 错了，因为 Y" > "X 可能值得商榷"
- 承认不确定处，偶尔自嘲降低距离感（"我的棋艺烂到看不懂定式"），但频率很低，点到为止
- 不用祈使句指挥读者（"快去试试"、"一定要记住"）
- 不卖货腔、不鸡汤、不导师腔

### 1.4 独特视角：只有 Ody 写得出的角度

每篇至少有一处"这个角度我没想过"。借外部素材（a16z 报告、Howard Marks 备忘录、创始人访谈）+ Ody 作为 AI-native Builder 的一手验证 = Ody 的声音。

纯转述 = 文章摘要，没有 Ody。必须穿插个人经验、选择、判断。

## 二、读起来流畅（手机扫读不卡壳）

流畅 = 节奏对 + 逻辑不断 + 没有废话打断。读者从头到尾不需要回头理解。

### 2.1 段落极短，节奏交错

- 1-3 句 / 段，经常一句话一段
- 长短句交错，连续 3 句长度相近 = 节奏呆板
- 至少 2-3 次一句话独立成段制造"断裂"
- 偶尔用疑问句做节奏刹车

### 2.2 开头前 3 句决定留存

第一句直接进入，零铺垫。三种切入：

- **数据/事实**："1997 年有个学者写了一本关于硬盘行业的书。28 年后，这本书的预言正在 AI 行业重演"
- **反常识**："Notion 联合创始人是团队里最强的程序员。但他已经很久不写代码了"
- **观察**："最近在看 Howard Marks 的最新备忘录，他和 Gavin Baker 对同一个问题的判断完全相反"

第 2-3 句建立"为什么值得读下去"。

禁止："在当今 AI 快速发展的时代..."、"随着技术的演进..."、"众所周知..."

### 2.3 逻辑链不断

- 偏离主线要有扣主线句拉回来
- 一个意思一个段落，不混装
- 重点加粗，扫读也能 get 核心判断
- **换行规则**：正文内句子之间、### 标题前、--- 分割线前后、[配图:] 标注之间一律单换行（\n），不出现空行（\n\n）。小红书平台以单换行为准

### 2.4 砍废话的标准

删掉这句话后，下一句是否依然成立？如果成立，这句就是冗余。

典型废话：
- 引入句："先说一个容易被忽视的事实"
- 过渡句："这个原则说起来简单，做到很难："
- 解释显而易见："没有人想用一个不停反驳你的产品"
- 装饰性比喻（只为了好看不帮助理解）
- 生硬过渡词："说白了"、"讲真"、"重点来了"

### 2.5 结尾回扣开头

- 开头埋的钩子/意象，结尾有变体回来（契诃夫之枪开火）
- 可以用开放性问题收尾引发讨论，前提是全文立场已清晰
- 不做总结性鸡汤
- 好的结尾示例："我倾向认为 X，原因是 Y。但 Z 值得持续关注"（有立场，有理由，有留白）

## 三、信息密度高（同样字数塞更多事实/判断）

Ody 的读者是 builder、投资人、从业者。他们要的是认知增量，不是情绪渲染。

### 3.1 每个观点必须有支撑

权威引用 + 真实数据 + 具体案例 + 跨行业类比，任选其一。空口论断直接砍。

- 用具体数字：不写"显著增长"，写"$10B+ 交易量"
- 不写"18 个月"如果真相是"6 个月"
- 不用"某客户"、"某朋友"——要么具体要么删

### 3.2 对比出认知差

- A vs B（Howard Marks vs Gavin Baker 对立判断）
- before / after
- 旧共识 vs 新现实
- 用旧行业类比新行业（硬盘行业 → AI 竞赛）
- 反常识具体案例（Notion 联创不写代码了）

### 3.3 预判质疑，不防御

读者可能会质疑什么？在文中回应，但不要像在辩解。"我非常理解这种感觉，但..."比直接驳斥更有说服力。

### 3.4 知识要"顺手掏出来"

引用（论文/书/历史）自然融入，不像"我特意去查的"。不是"下面我来科普一下 X"，而是"这让我想起 X"。

### 3.5 术语处理

- 本土化：Slack → 飞书
- 关键英文术语保留 + 中文标注：MCP（Model Context Protocol，模型上下文协议）
- 大白话优先："前沿公司" → "AI 大厂"；"人类天然知道" → "我们知道"；"是它的反面" → "是累赘"
- 受众比程序员更广，非技术读者也要能跟上

### 3.6 信息密度 vs 情感渲染

同样字数里，宁可多一个事实，也不多一句情感渲染。

- 少文学性多信息量
- 少感受多判断

## 硬性规则（违反就改回来）

- 不用 emoji
- 不用感叹号
- 不用祈使句
- 不用破折号（用冒号替代）
- **不用"不是 X，是 Y"句式**（所有变体：不是 X 而是 Y / 不是 X，是 Y / 不只是 X，更是 Y）——无论 X Y 多短，一律重写为正向陈述
- 不用博主腔（家人们、姐妹们、宝子们、绝绝子、yyds）
- 不用情绪化表达（真的太绝了、强烈推荐、闭眼入）
- 不用导师腔（你应该、一定要、记住）
- 不用绝对化用语（最好、第一、100%）
- 不滥用连接词（然而、事实上、与此同时、从某种意义上——保留一半以下）
- 不用教科书结构（首先...其次...最后）
- 不用翻译腔动词堆（作为...关于...基于...进行...）
- 不用"本质上、归根结底、换句话说、值得注意的是、不难发现、综上所述"

### 段落内粗体短标题

粗体短标题（如 **谁最危险**、**关键原则**）不加句号。

---

## 标题与封面

- 标题信息密度高，直接陈述观点或制造认知冲突
- 不用 emoji，不用感叹号
- 封面通常白底黑字，和标题风格一致

两种高点击模式：
- **经典梗 + 反问**："这次不一样，这句话熟悉吗"
- **A vs B 对立 / 时间差悬念**："1997 年就预言了 AI 竞赛"

## 自查清单（写完后过一遍）

- [ ] 活人感：有没有 AI 自己编造的第一人称细节？有没有标签式"我的框架是"？
- [ ] 流畅：前 3 句能独立抓住注意力？长短句有交错？有地方要回头理解逻辑吗？
- [ ] 信息密度：每个观点有具体支撑？有没有情感渲染占了信息位置？
- [ ] 硬性规则：emoji、感叹号、祈使句、破折号、"不是 X 是 Y"——全部零命中？
- [ ] 立场：读者读完知道 Ody 倾向哪一边吗？还是"两边都有道理"和稀泥？
