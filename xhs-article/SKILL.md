# 小红书笔记创作

Ody 的个人 IP 是"AI 全栈 Builder，把点子变成现实的人"。不是博主，是做个人品牌，注重输出质量。内容定位是认知深度和信息密度，不是情绪刺激和流量套路。

**本 Skill 由 `/create` 路由调用，Phase 编号与 create 统一。** Phase 0-1（选题 + 标题）和 Phase 5（AI 扫描 + 终审）由 create 处理。本 Skill 负责 Phase 2、3、4、6。

Phase 2 完成后等 Ody 确认。Phase 3 和 Phase 4 连续执行，中途不停，唯一例外是 Phase 4 的中英文对照表需要 Ody 确认后再统一修改。

## Phase 2: 结构与大纲

角度确认后：

1. **结构选型** — 根据内容选择结构模式：
   - **"N 点并列洞察"**：适合行业分析帖，每个分论点是"观点 → 案例 → 类比 → 短句收尾"，小标题用直接陈述而非修辞句式
   - **"线性论证"**：适合长文，一层推一层，"现象观察 → 路径梳理 → 战略分析 → 开放收尾"
2. **小标题** — 每个段落有清晰的信息点，小标题是直接陈述，不玩修辞。**小标题不加序号**（不用"一、二、三"或"1. 2. 3."），直接写内容本身
3. **素材储备** — 每个要点至少配一个：权威引用、真实数据、对比案例、跨行业类比
4. **信息核实** — 定稿大纲前，上网搜索核实（优先使用 2026 年最新信息）：
   - 提到的产品名称、版本、功能是否最新（2026 年 2 月的最新状态）
   - 数据和统计是否准确、来源可靠、时效性如何（优先 2025-2026 年数据）
   - 案例和故事是否属实、时间线是否正确
   - 引用的人物观点是否准确归属、是否为最新表态
   - **避免使用过时案例**：如果某个产品/公司/趋势在 2026 年已经过时或被取代，必须更新
5. **配图规划** — 正文配图（流程图、对比图、框架图等）：
   - 正文配图用 `[配图: 描述]` 标注在正文中对应位置
   - 配图将在 Phase 6 排版时用 HTML/CSS 内嵌到正文页中，与文字同页排版。规划时需考虑：图表类型（对比图、流程图、数据卡片）、包含的具体元素、哪些元素需要红色强调
   - 每篇至少规划 1 张信息图，将核心框架或对比可视化
6. **开头** — 规范详见 `references/style-guide.md` 的「开头」部分
7. **结尾** — 可以用开放性问题收尾引发思考，但全文必须有明确的判断和立场。不做"两边都有道理"的骑墙派

呈交完整大纲（含素材和配图规划），等 Ody 确认。

## Phase 3 + 4: 撰写初稿与审稿（连续执行，不中断）

Phase 3 初稿写完后，**直接进入 Phase 4 审稿，不停下来询问 Ody**。唯一需要暂停等待确认的是：Phase 4 第 5 步的中英文对照表。

## Phase 3: 撰写初稿

详细写作规范见 references/style-guide.md。

**核心要求：**
- Ody 的声音：谦逊但有立场的观察者，不是导师。个人观点要自然融入论述，像朋友聊天时顺口说出来的判断，而非生硬插入的"我的框架是：""我的观察是：""我的判断："。好的写法：用具体的个人选择或经验带出立场（如"我宁愿承担亏掉30%资产的风险，也不愿意错过浪潮"），让读者通过你的行动和选择感受到你的判断，而非通过标签式的声明
- 段落极短：1-3 句/段，短句独立成段制造节奏感
- **换行规则**：正文内所有内容（句子、### 标题、--- 分割线、[配图:] 标注）之间一律只用单换行（\n），不出现任何空行（\n\n）。无需遵守 Markdown 标准格式，小红书平台以单换行为准。
- **粗体短标题**：段落内的粗体短标题（如 **谁最危险**、**关键原则**）不加句号
- 重点加粗，让扫读也能 get 核心判断
- 每个观点必须有信息支撑（数据、案例、权威引用），不空喊口号
- **信息密度优先**：少文学性多信息量，少感受多判断
- **时效性优先**：所有数据、案例、产品功能必须是 2026 年最新状态。如果引用历史案例，明确标注时间（如"2023 年 GPT-4 发布时..."）
- 用 `[配图: 描述]` 标注配图位置
- 结尾可以用开放性问题引发讨论，但全文立场必须清晰
- **简介引言与正文同步生成**：简介引言用 1-3 句话概括文章的中心思考，让读者判断值不值得读。**必须以具体例子开头**（Ody 自己的成果、别人的案例、一个具体事实），然后才是方法论关键词或核心心法。绝对不能是抽象原则的罗列。写完正文后立即写简介引言，一起输出到文件

**Ody 的写法特征：**
- 呈现对立判断但不急于站队（Howard Marks vs Gavin Baker）
- 用旧行业类比新行业（硬盘行业预言 AI 竞赛）
- 反常识的具体案例做论据（Notion 联创是最强程序员但不再写代码）
- 术语本土化（Slack → 飞书），关键英文术语保留加中文标注（MCP/Model Context Protocol，模型上下文协议）
- 偶尔自嘲降低距离感，但频率很低，点到为止
- **基于外部素材时，必须穿插 Ody 的个人经验或判断**，把"转述"变成"验证"，避免写成纯粹的文章摘要。但插入方式要自然：用具体的个人行动、选择、经历带出观点（"每隔几个月回头看之前的做法都觉得原始""我宁愿多花时间在X上，也不想Y"），避免"我的框架是：""我的判断："这种生硬的标签式声明

**砍冗余的标准：**
- "正确但不推进论点"的句子一律删掉。典型的废话模式：引入句（"先说一个容易被忽视的事实。"）、过渡句（"这个原则说起来简单，做到很难："）、解释显而易见的事情（"没有人想用一个不停反驳你的产品。"）
- 判断标准：删掉这句话后，下一句是否依然成立？如果成立，这句就是冗余
- **措辞偏口语、偏直接**："前沿公司" → "AI大厂"，"人类天然知道" → "我们知道"，"是它的反面" → "是累赘"。能用大白话说的不用书面语

## Phase 4: 审稿与打磨

初稿完成后：

1. **简介引言审查** — 简介引言是否准确概括了文章的中心思考？和正文立场是否一致？是否信息密度够高、不空洞？
2. **删装饰性修辞** — "聪明但不增加信息量"的句子砍掉。比喻如果只是为了好看而不是帮助理解，直接删。信任读者自己得出结论，不把话说透
2. **加判断锐度** — 模糊表述换成更尖锐的判断。加入更精确的概念，加英文标注给需要的读者
3. **信息密度检查** — 同样字数里能不能多塞一个事实或概念？宁可多一个信息点，也不多一句情感渲染
4. **硬性规则核查** — 逐条检查：没有 emoji，没有感叹号，没有祈使句，没有破折号，没有多余的过渡语气词，**严禁所有"不是X...是Y"句型变体**（包括：不是X，而是Y / 不是X，是Y / 不只是X，更是Y / 不是X而是Y）——无论是否有逗号，无论X和Y多短，一律重写为正向陈述
5. **中英文用词审查** — 扫描全文所有英文词汇，输出一张对照表给 Ody 确认后再改：
   - **保留英文**列：真正的专业术语（如 Agent、LLM、CI/CD、orchestrator、sub-agent、token、reward shaping 等），首次出现加中文标注
   - **改为中文**列：有自然中文对应的非专业词汇（如 session → 会话、spawn → 启动、issue → 任务、PM → 项目经理、input/output → 输入/输出）
   - 判断标准：如果一个词在中文技术社区已经是通用说法（如 Agent、token、prompt），保留；如果只是英文日常词汇混入中文句子，换成中文
   - **等 Ody 确认对照表后再统一修改**，不要自行决定
6. **事实核查** — 数据准不准？案例对不对？上网再确认一遍 Phase 2 中没核实的内容
6. **配图检查** — 正文中 `[配图: 描述]` 标注到位了吗？每个配图描述是否包含足够信息（图表类型、具体元素、强调项）供 Phase 6 排版时直接实现？
7. **平台合规** — 检查绝对化用语（"最好""第一""100%"）和敏感词，避免被限流
8. **标签规划** — 准备 5-10 个相关话题标签（#xxx），混搭大流量标签和精准长尾标签

## Phase 6: 报纸风排版与截图

Phase 4 完成、中英文对照表确认后，回到 create Phase 5 做 AI 扫描 + 终审。终审通过后进入本阶段。

### 6.1 生成封面（等 Ody 确认）

使用封面模板脚本生成封面，不要每次手写 HTML。模板在 `scripts/cover_template.html`，通过 `scripts/generate_cover.py` 填入文字并截图：

```bash
python3 ~/.claude/skills/xhs-article/scripts/generate_cover.py \
  --date "2026-04-13" \
  --source "来源人物 · 身份" \
  --number "98" \
  --unit "%" \
  --data-line '<em>uptime</em> · 成了行业新常态' \
  --title '主标题第一行<br>主标题第二行' \
  --subtitle '副标题（金句或认知冲突）' \
  --output /Users/odyzhou/Desktop/xhs-posts/covers/page-01.png
```

封面设计要点：
- **核心数字做视觉锚点**（最大字号，缩略图可读），数字必须与文章内容有逻辑关联
- 主标题精简到 1-2 行
- 副标题用原文金句（有认知冲突的引用）
- **日期默认用发布日（今天），不是素材原文日期**

封面字号和配色详见 `references/newspaper-style-template.md`。如需调整模板（如字号、布局），直接修改 `cover_template.html`。

**打开截图给 Ody 确认封面效果后再往下走。**

### 6.2 生成全文排版页（连续执行）

封面确认后，参考 `scripts/generate_newspaper_pages.py` 的模版结构，为当前文章生成全部正文页：

**分页原则：**
- 每页正文约 350-400 字（38px 字号下的可读容量）
- 新章节标题开启新页或在上一页自然衔接
- 最后一页为居中排版的结尾金句
- 末页为参考文献

**正文页设计规范：**
- 同封面的牛皮纸背景、噪点纹理、装饰性分栏线
- 页眉：左 "DEEP DIVE"，右 页码 "03 / 11"
- 章节标题：Noto Serif SC, 52px, 900
- 正文：Noto Serif SC, 38px, 400, line-height 1.85
- 加粗：weight 700, color #111
- 页脚：左 "Ody · [来源]"，右 装饰性大页码
- container padding: 48px top, 80px sides, **120px bottom**（防止底部被截）

### 6.3 内嵌信息图（默认内嵌，非可选）

文章中标注了 `[配图: 描述]` 的位置，**默认将信息图内嵌到对应正文页中**，与文字同页排版。只有当页面空间确实不够时，才单独成页。

适合内嵌的类型：
- **对比框架图**（如传统 vs 新模式流程对比）
- **流程/演进图**（节点 + 箭头）
- **数据卡片**（关键数字突出展示）
- **引用卡片**（金句 + 来源，报纸 pull quote 风格）

做法：在对应页的 HTML 中，用 CSS 绘制图表元素（flex 布局、border、background 等），保持报纸风格的配色和字体。不依赖外部图片资源，纯 HTML/CSS 实现。信息图用 `#b91c1c` 红色做强调色标记关键元素。

### 6.4 自查截图（必须在给 Ody 看之前完成）

**所有页面截图完成后，必须逐页 Read PNG 自查，确认以下项目无误后再呈交 Ody：**

1. **无截断/溢出** — 文字没有被底部截掉，内容在页面可视范围内
2. **加粗渲染正常** — `<strong>` 标签生效，粗体清晰可辨
3. **信息图完整** — 图表元素（框、箭头、标签）全部渲染，无缺失
4. **字体加载成功** — 中文字体（Noto Serif SC）正确显示，非 fallback 字体
5. **CJK smart quote 问题** — 检查英文撇号是否有多余空格（Noto Serif SC 中 `&rsquo;` 会渲染异常，应使用 ASCII 撇号 `'`）
6. **页码连续** — 页眉页码 "02/10" 格式正确，与实际总页数一致

发现问题时当场修 HTML 并重新截图，不要把问题留给 Ody。

**生成流程汇总：**
1. 用 Python 脚本定义每页内容，批量生成 HTML
2. Chrome headless 批量截图为 PNG
3. **逐张 Read PNG 自查**（上述 6 项）
4. 修复发现的问题，重新截图
5. 全部通过后一次性呈交 Ody

所有 PNG 输出到 `/Users/odyzhou/Desktop/xhs-posts/covers/`，文件名 `page-01.png` 到 `page-NN.png`，按顺序上传小红书。

### 6.5 HTML 编写注意事项

- **避免 `&rsquo;` `&lsquo;` 等 smart quote HTML 实体**：在 Noto Serif SC 正文中会渲染出多余空格。直接用 ASCII 撇号 `'` 和普通引号 `"`。`&ldquo;` `&rdquo;` 在中文语境下可用（渲染为中文引号""）
- **长标题用 `<br>` 换行**：section-title 在 52px 下一行约 17 个中文字符，超出时手动加 `<br>` 控制断行位置
- **pull-quote 用 Playfair Display**：英文金句用 pull-quote 样式（Playfair Display italic），视觉区分度更高，且该字体的 smart quote 渲染正常
