# Platform: xiaohongshu (小红书)

xhs 平台规格 + 平台特化写作约束 + Phase 1-6 执行细节。是 [[voice]] 之上的加成，包含 voice 之外所有 xhs 特化的执行流程。

`/create` 在 target=xhs 时按本文件 Phase 1-6 章节执行。

---

## 规格

- **文案上限**：~1000 字（取决于排版页大小）
- **标题**：主 + 副组合，主标题 ≤ 14 字，副标题 ≤ 20 字
- **配图**：1-9 张，封面权重最高
- **算法核心**：完读率 + 收藏 + 标签匹配
- **节奏**：周更 1-2 篇深度图文，日更短图文可选

---

## 平台特化写作约束（voice.md 之上的加成）

### 段落极短，节奏交错

- 1-3 句 / 段，经常一句话一段
- 长短句交错，连续 3 句长度相近 = 节奏呆板
- 至少 2-3 次一句话独立成段制造"断裂"
- 偶尔用疑问句做节奏刹车

> 这条是 xhs 特化（图文渲染 + 手机扫读特性），不适用于 X 长推或 LinkedIn 长帖。

### 换行规则

正文内句子之间、`###` 标题前、`---` 分割线前后、`[配图:]` 标注之间一律**单换行**（`\n`），不出现空行（`\n\n`）。

小红书平台以单换行为准，不遵守 Markdown 标准格式。

### 粗体短标题

段落内的粗体短标题（如 **谁最危险**、**关键原则**）不加句号。

## Phase 1：标题（xhs）

### 双标题模式

- 主标题：≤ 14 字，抓人 / 制造悬念
- 副标题：≤ 20 字，给观点密度 / 金句 / 立场
- **副标题必须提供主标题没说的信息**，不能只是扩写
- 判定：删掉副标题完全不损失信息 = 失败

### 标题 5 条铁律

1. 留悬念不说答案（用"为什么"不用"证明"）
2. 话题扩大不缩窄（普世词替换行业术语）
3. 击中真实痛点（"靠兴趣赚不到钱" > "消费型兴趣"）
4. 张力检查至少命中 2 项（对比 / 数字 / 悬念 / 冲突 / 时间 / 结果承诺）
5. ≤ 20 字含标点

### 两种高点击模式

- **经典梗 + 反问**："这次不一样，这句话熟悉吗"
- **A vs B 对立 / 时间差悬念**："1997 年就预言了 AI 竞赛"

### 标题生成流程

1. 判断话题属性 → 锁定 12 类触发器中对应的 2-3 类（认知冲突 / 好奇缺口 / 恐惧损失 / 身份代入 / 数字锚定 / 结果承诺 / 社会证明 / 争议挑衅 / 场景条件 / 行动号召 / 权威借力 / 互动测试）
2. 选 5-8 个公式，至少覆盖 3 种触发器类型
3. 套入话题生成主标题候选，每个标注公式编号 + 原始爆款对照 + 推荐理由
4. 每个主标题配 2-3 个副标题候选
5. 输出 Top 3 主+副组合，说明配合逻辑

> **75 公式全量 + 12 类触发器详解 + 多行业举例** → `references/xhs-titles.md`

### Ody 风格过滤

公式库中偏营销/鸡汤/情绪化的公式套用时过滤：

- 不用 emoji、感叹号、博主腔、绝对化用语
- 保持认知冲突和信息密度导向
- 谦逊观察者定位：导师腔公式改写（"给你的终极建议" → "我观察到的 N 个规律"）

### 封面

- 通常白底黑字，和标题风格一致
- 不用 emoji，不用感叹号（与 voice.md 一致）
- 数字锚点 + 主标题 + 副标题三层

## Phase 2：大纲（xhs）

角度确认后（结构由 Phase 0.3 的原型决定，这里不再做结构选型）：

### 2.1 小标题

每个段落有清晰的信息点，小标题是直接陈述，不玩修辞。**小标题不加序号**（不用"一、二、三"或"1. 2. 3."），直接写内容本身。

### 2.2 素材储备

每个要点至少配一个：权威引用、真实数据、对比案例、跨行业类比。

### 2.3 信息核实

定稿大纲前，上网搜索核实（优先使用 2026 年最新信息）：

- 提到的产品名称、版本、功能是否最新（2026 年的最新状态）
- 数据和统计是否准确、来源可靠、时效性如何（优先 2025-2026 年数据）
- 案例和故事是否属实、时间线是否正确
- 引用的人物观点是否准确归属、是否为最新表态
- **避免使用过时案例**：如果某个产品/公司/趋势在 2026 年已经过时或被取代，必须更新

### 2.4 配图规划

默认姿态是**不配图**。只有当文字力所不及时才提出配图。

⚠️ **AI 产品落地分析型例外**：此原型默认配 **3 张外部真实图**（创始人头像 + 产品 UI 截图 + 团队/收束照），不走下面的两道门。详见 `_schema/prototypes.md`「配图规则（分析型）」段。Phase 2 大纲必须列 3 张图的位置 + 描述 + 来源 URL，并在 Phase 6 之前下载到 `~/Desktop/xhs-build/[slug]/images/`。Heidi 篇是这种配图模板的标杆参考（见 `scripts/xhs-render/generate_heidi_pages.py` 的 TOM_KELLY / PHONE_UI / TEAM 三个 img_b64）。

**提案前自检（两道门，必须都过）**：

- **第一道门：文字够了吗？** 把这段文字读一遍，读者真的需要图才能理解吗？如果不需要就不配
- **第二道门：这张图只能靠图才成立吗？** 必须满足以下之一，否则砍掉提案：
  - **信息增量**：图里有文字没讲过的信息（额外数据、隐含结构、层级嵌套、多维交叉）
  - **辅助理解**：图把文字中抽象的关系变成可视的（流程先后、空间位置、节点连接、时间演进、嵌套层级）

**以下模式是伪信息图，提案阶段就禁止出现**：

- ❌ 两个并排卡片做"A vs B"对比（当文字已经列出了维度和案例，图就是复述）
- ❌ N 个卡片复述文字里的 N 条列表
- ❌ 数据卡片只为"有图感"（文字里没强调过的数字，硬放大显示）
- ❌ 定义标签卡片（把概念定义原文放进方框里）
- ❌ 框架图只有文字已经写清的两个名词并排

**真正该做的配图类型（示例）**：

- ✅ 时间演进轴（第 1 年 → 第 2 年 → 第 3 年 + 状态变化）
- ✅ 分支结构图（一个起点分出多条路径，每条路径的终点不同）
- ✅ 节点连接图（多方关系、数据流向、依赖顺序）
- ✅ 叠加层级图（多层嵌套关系，盖图会丢失结构）
- ✅ 补充数据可视化（文字没提的具体数据变成图表）

**配图标注**：用 `[配图: 描述]` 放在正文对应位置。Phase 6 用 HTML/CSS 内嵌到正文页。描述要写清：图表类型、具体元素、哪些元素用红色强调。

**总原则**：宁可一张图都没有，不要为了"有图感"硬凑。每篇不设配图数量目标。Phase 2 提案、Phase 4 复审、Phase 6 终审，三道 gate 每道都可以砍。

**再次强调（分析型反例教训）**：写 AI 产品落地分析型时，「默认不配」会让 11-14 页全是文字 + 卡片 + 表格，视觉上比 Heidi 篇枯燥。分析型走自己的配图规则，不走这两道门。

### 2.5 开头

规范见 `_schema/voice.md` §3.1（三种切入：数据/反常识/观察）。深度优化方法见 `references/hooks.md`。

### 2.6 结尾

可以用开放性问题收尾引发思考，但全文必须有明确的判断和立场。不做"两边都有道理"的骑墙派。

呈交完整大纲（含素材和配图规划），等 Ody 确认。

## Phase 3：撰写初稿（xhs）

写作规范以 `_schema/voice.md` + 本文「平台特化写作约束」为准。下面只列 Phase 3 执行时的**操作要点**，不重复规则。

**操作要点：**

- **时效性**：所有数据、案例、产品功能必须是 2026 年最新状态。引用历史案例明确标注时间（如"2023 年 GPT-4 发布时..."）
- **配图标注**：用 `[配图: 描述]` 放在正文对应位置（Phase 6 用 HTML/CSS 内嵌实现）
- **简介引言与正文同步生成**：1-3 句话概括文章中心思考。**必须以具体例子开头**（Ody 自己的成果、别人的案例、一个具体事实），然后才是方法论关键词或核心心法。不能是抽象原则的罗列。写完正文立即写简介引言，一起输出到文件

**Phase 3 初稿写完后直接进入 Phase 4，不停下来询问 Ody**。唯一需要暂停等待确认的是 Phase 4 的中英文对照表。

## Phase 4：审稿与打磨（xhs）

初稿完成后逐条执行：

1. **简介引言审查** — 简介引言是否准确概括了文章的中心思考？和正文立场是否一致？是否信息密度够高、不空洞？
2. **删装饰性修辞** — "聪明但不增加信息量"的句子砍掉。比喻如果只是为了好看而不是帮助理解，直接删（对照 voice.md §6.4 比喻必要性）
3. **加判断锐度** — 模糊表述换成更尖锐的判断。加入更精确的概念，加英文标注给需要的读者
4. **信息密度检查** — 同样字数里能不能多塞一个事实或概念？宁可多一个信息点，也不多一句情感渲染
5. **硬性规则核查** — 对照 `_schema/voice.md` §5（硬性规则）+ §6（句子要直接）全文扫描，零命中通过
6. **中英文用词审查** — 扫描全文所有英文词汇，输出一张对照表给 Ody 确认后再改：
   - **保留英文**列：真正的专业术语（如 Agent、LLM、CI/CD、orchestrator、sub-agent、token、reward shaping 等），首次出现加中文标注
   - **改为中文**列：有自然中文对应的非专业词汇（如 session → 会话、spawn → 启动、issue → 任务、PM → 项目经理、input/output → 输入/输出）
   - 判断标准：如果一个词在中文技术社区已经是通用说法（如 Agent、token、prompt），保留；如果只是英文日常词汇混入中文句子，换成中文
   - **等 Ody 确认对照表后再统一修改**，不要自行决定
7. **事实核查** — 数据准不准？案例对不对？上网再确认一遍 Phase 2 中没核实的内容
8. **配图检查** — 两步走：
   - **先判去留**：每张 `[配图: ...]` 盖住，读者理解有损失吗？无损失 = 砍掉
   - **再看描述**：保留下来的配图，描述是否包含足够信息（图表类型、具体元素、强调项）供 Phase 6 排版时直接实现？
9. **平台合规** — 检查绝对化用语（"最好""第一""100%"）和敏感词，避免被限流
10. **标签规划** — 准备 5-10 个相关话题标签（#xxx），混搭大流量标签和精准长尾标签

## Phase 6：报纸风排版与截图（xhs）

Phase 4 完成、中英文对照表确认后，回到 `/create` Phase 5 做 AI 扫描 + 终审。终审通过后进入本阶段。

### 6.1 生成封面（等 Ody 确认）

使用封面模板脚本生成封面，不要每次手写 HTML。模板在 `scripts/xhs-render/cover_template.html`，通过 `scripts/xhs-render/generate_cover.py` 填入文字并截图：

```bash
python3 ~/.claude/skills/create/scripts/xhs-render/generate_cover.py \
  --date "2026-04-13" \
  --source "来源人物 · 身份" \
  --number "98" \
  --unit "%" \
  --data-line '<em>uptime</em> · 成了行业新常态' \
  --title '主标题第一行<br>主标题第二行' \
  --subtitle '副标题（金句或认知冲突）' \
  --output /Users/odyzhou/Desktop/xhs-build/[slug]/page-01.png
```

封面设计要点：
- **核心数字做视觉锚点**（最大字号，缩略图可读），数字必须与文章内容有逻辑关联
- 主标题精简到 1-2 行
- 副标题用原文金句（有认知冲突的引用）
- **日期默认用发布日（今天），不是素材原文日期**

封面字号和配色详见 `scripts/xhs-render/newspaper-style-template.md`。如需调整模板（如字号、布局），直接修改 `cover_template.html`。

**打开截图给 Ody 确认封面效果后再往下走。**

### 6.2 生成全文排版页（连续执行）

封面确认后，参考 `scripts/xhs-render/generate_newspaper_pages.py` 的模版结构，为当前文章生成全部正文页：

**分页原则：**
- 每页正文约 350-400 字（38px 字号下的可读容量）
- 新章节标题开启新页或在上一页自然衔接
- 最后一页为居中排版的结尾金句
- **末页为参考文献页（必做，不得省略）**

#### 末页参考资料页（强制）

**为什么强制**：xhs 帖子是公开发布物。每篇文章的事实/数字/引文都需要可追溯来源。漏这一页 = 内容失去可验证性。Ody 反复在终审时发现这页被漏，所以这里固化为不可跳过步骤。

**触发条件**：只要文章正文 markdown 末尾有 `参考资料：[URL...]` 段落或正文中引用了任何外部数据/案例/原文片段（数字、引文、产品名背书），最后一页必须是参考资料页。

**唯一例外**：文章是纯 Ody 一手随笔，零外部素材。这种情况在 Phase 4 输出末必须显式声明"无参考资料"，否则视为漏做。

**参考资料页内容结构（按这个填）**：
- 章节标题：`参考资料`
- 主条目：来源名 · 日期 + 文章/视频/Podcast 标题 + URL（URL 用 32px 浅灰色显示）
- 多个来源每条独立一段
- 底部加分隔线 + 免责说明（信源说明 + 观点声明）

**参考脚本**：`scripts/xhs-render/generate_harness_blog_pages.py` 的 page-11 块是标准实现，复用即可。

**TOTAL 页数核算**：参考资料页计入 TOTAL。例如正文 8 页 + 参考资料 1 页 + 封面 1 页 = TOTAL 10。改 TOTAL 后所有页码自动更新（脚本里所有 make_page 调用都用 TOTAL 变量）。

#### 正文页设计规范

- 同封面的牛皮纸背景、噪点纹理、装饰性分栏线
- 页眉：左 "DEEP DIVE"，右 页码 "03 / 11"
- 章节标题：Songti SC, 52px, 900
- 正文：Songti SC, 38px, 400, line-height 1.85
- 加粗：weight 700, color #111
- 页脚：左 "Ody · [来源]"，右 装饰性大页码
- container padding: 48px top, 80px sides, **120px bottom**（防止底部被截）

**字体策略（重要，不要改回 Google Fonts）**：全部用 macOS 本地字体——中文衬线 `Songti SC`、中文无衬线 `PingFang SC`、英文衬线/装饰 `Georgia`。**禁止** `@import url('https://fonts.googleapis.com/...')`——国内网络不通时每页 Chrome 等字体 25 秒+，13 页可能跑半小时。本地字体跑完 13 页 ≈ 27 秒。

**清晰度策略（重要）**：Chrome 截图 flag 必须包含 `--force-device-scale-factor=2`，输出 2160×3600 的 2x retina PNG。XHS 上传后保留更多细节，用户放大看官方架构图时不糊。已写进 `generate_*_pages.py` 模板。

**内嵌外部图片（OpenAI / Anthropic blog 截图等）的清晰度规则**：
1. **用全分辨率原图**，不要用 `-small` 后缀的压缩版（截图比 web 显示分辨率高，base64 内联即可，文件大小不是问题）
2. CSS 中 `figure-img-wrap img { max-width: 920px }` 只是 CSS 约束；浏览器在 2x DPI 下会用 1840px 渲染，所以源图实际像素 ≥ 1840px 才不糊
3. 上线前 Read 一下生成的 PNG，放大看图内文字能否清晰

### 6.3 内嵌信息图（只嵌有信息增量的）

文章中标注了 `[配图: 描述]` 的位置，**排版前再过一次图文互补检查**。Phase 4 已经砍过一轮，这里做最后一道 gate：

**保留标准**（必须满足其一，否则砍）：
- **信息增量**：图里有文字没讲过的信息（额外数据、结构、维度）
- **辅助理解**：图把文字中抽象的关系变成可视的（流程先后、空间位置、层级嵌套、元素交互）

**常见该砍的伪信息图**：
- 把文字里的"A vs B"复述成两个并排方框（同样四条列表排两遍不叫对比图）
- 把文字里的 N 条列表复述成 N 个卡片
- 把已经说清的概念定义再写一遍当标签

**保留下来的类型**：
- **对比框架图**（必须有文字没给的维度，不是把两列简单并排）
- **流程/演进图**（节点 + 箭头，有时间或因果结构）
- **数据卡片**（关键数字突出展示，文字里没单独强调的）
- **引用卡片**（金句 + 来源，报纸 pull quote 风格）

**做法**：在对应页的 HTML 中，用 CSS 绘制图表元素（flex 布局、border、background 等），保持报纸风格的配色和字体。不依赖外部图片资源，纯 HTML/CSS 实现。信息图用 `#b91c1c` 红色做强调色标记关键元素。

### 6.5 HTML 编写注意事项

- **避免 `&rsquo;` `&lsquo;` 等 smart quote HTML 实体**：在 Songti SC 正文中会渲染出多余空格。直接用 ASCII 撇号 `'` 和普通引号 `"`。`&ldquo;` `&rdquo;` 在中文语境下可用（渲染为中文引号""）
- **长标题用 `<br>` 换行**：section-title 在 52px 下一行约 17 个中文字符，超出时手动加 `<br>` 控制断行位置
- **pull-quote 用 Georgia**：英文金句用 pull-quote 样式（Georgia italic），视觉区分度更高，且该字体的 smart quote 渲染正常
